# 8.0 Fidelity Checklist & Epic Backlog

## 1. Purpose

This document defines the delivery plan to make this multiplayer game match the feel and rules of **Digimon Digital Card Battle (PS1)** as closely as possible.

It has two goals:

1. Provide a **fidelity checklist** that can be reviewed as pass/fail.
2. Break the work into **epics and tasks** that can be completed one by one.

---

## 2. Canonical references (source of truth)

Use these references in priority order when deciding behavior:

1. PS1 manual + verified gameplay behavior (battle flow, support/effect rules, phase order).
2. Card-level behavior references (card text/effect catalogs).
3. Multi-match gameplay videos for pacing and UX timing validation.

Notes:

- A single video is not enough for mechanical parity.
- When references disagree, document the chosen behavior in the task that implements it.

---

## 4. Epics and task breakdown (resolve one by one)

## Epic E0 - Rule Contract and Baseline

**Goal:** Remove ambiguity before implementation.
**Status:** Complete

Tasks:

- [x] T0-1: Create `docs/fidelity-rules-contract.md` with authoritative per-rule statements and reference citation per rule.
- [x] T0-2: Add a "resolved ambiguities" section for any conflicting references.
- [x] T0-3: Freeze a v1 scope list (in-scope vs deferred).

Exit criteria:

- Every fidelity checklist item maps to a written rule statement.

Dependencies:

- None.

---

## Epic E1 - Data Model Expansion (Card Kinds + Effect IDs)

**Goal:** Enable missing mechanics without ad-hoc logic.

Tasks:

- [x] T1-1: Extend card schema with `cardKind` (`digimon`, `option`, `evolution_option`).
- [ ] T1-2: Add normalized `effectId` + `effectArgs`.
- [ ] T1-3: Add migration/loader for current card JSON format.
- [ ] T1-4: Validate unknown effect IDs at startup and fail fast.

Exit criteria:

- Server can parse and validate full card taxonomy and effect descriptors.

Dependencies:

- E0.

---

## Epic E2 - Opening Flow and Preparation Parity

**Goal:** Align opening setup and preparation with source behavior.

Tasks:

- T2-1: Implement opening hand size and redraw/mulligan rules.
- T2-2: Implement opening Digimon selection constraints and penalties.
- T2-3: Refine DP/evolution sequence to match selected contract.
- T2-4: Add preparation-phase validation logs for replay/audit.

Exit criteria:

- Opening and preparation phase outcomes match reference scenarios.

Dependencies:

- E0, E1.

---

## Epic E3 - Option and Evolution Option Execution

**Goal:** Introduce missing Option mechanics into authoritative runtime.

Tasks:

- T3-1: Add server action handlers for Option usage.
- T3-2: Implement evolution-option timing window and legality checks.
- T3-3: Add effect resolution hooks for option effects into phase machine.
- T3-4: Add UI affordances for option selection and state feedback.

Exit criteria:

- Option cards can be played and resolved according to rule contract.

Dependencies:

- E1, E2.

---

## Epic E4 - Battle Effect Engine (X logic + conflicts)

**Goal:** Match original battle depth and edge-case behavior.

Tasks:

- T4-1: Build effect interpreter for attack modifiers, counters, to-0, crash, life-steal.
- T4-2: Implement ordered effect pipeline with tie-break rules.
- T4-3: Add explicit conflict resolver for competing effects.
- T4-4: Cover double-KO and special cancellation edge cases.

Exit criteria:

- Cross and special effect interactions pass deterministic scenario suite.

Dependencies:

- E1, E3.

---

## Epic E5 - Support Phase Parity

**Goal:** Align support choices and reveal behavior.

Tasks:

- T5-1: Implement support selection order parity (attacker/defender ordering as contracted).
- T5-2: Add optional support draw from online deck logic (if in-scope).
- T5-3: Preserve hidden info and synchronize reveal deterministically.
- T5-4: Match support effect-speed ordering and tie behavior.

Exit criteria:

- Support flow and outcomes match reference checklist cases.

Dependencies:

- E3, E4.

---

## Epic E6 - Multiplayer Anti-Griefing

**Goal:** Ensure games cannot stall and are fair under latency/AFK.

Tasks:

- T6-1: Add server phase timers by phase type.
- T6-2: Add timeout auto-commit defaults.
- T6-3: Add consecutive inactivity strikes + forfeit.
- T6-4: Emit structured timeout/forfeit telemetry for audits.

Exit criteria:

- Matches always progress; AFK behavior is deterministic and enforceable.

Dependencies:

- E2 onward.

---

## Epic E7 - Deckbuilding and Match Configuration

**Goal:** Move from auto-generated decks to authentic constrained decks.

Tasks:

- T7-1: Implement deck validation API (size, copies, restricted cards).
- T7-2: Replace runtime auto-generated decks with player deck load.
- T7-3: Add support for arena rule variants/special rules.
- T7-4: Add invalid-deck rejection with clear error messages.

Exit criteria:

- Match starts only with legal decks and selected rule profile.

Dependencies:

- E1.

---

## Epic E8 - Deterministic Replay and AI Review Harness

**Goal:** Make multiplayer behavior easy to review without full UI automation.

Tasks:

- T8-1: Emit structured battle logs per phase (actions, effects, damage, state deltas).
- T8-2: Add seed-based deterministic replay mode for server simulation.
- T8-3: Add scenario tests for all critical fidelity IDs (FC-001..FC-030).
- T8-4: Add CI job to run scenario suite and publish failures by fidelity ID.

Exit criteria:

- AI agent can run replay + scenario suite and produce objective pass/fail reports.

Dependencies:

- E2 to E7.

---

## 5. Recommended execution order

1. E0 -> E1 -> E2
2. E3 + E4 + E5
3. E6 + E7
4. E8 (and then keep extending scenario coverage each release)

Why this order:

- It de-risks rule ambiguity first.
- It avoids rework by normalizing data before effect expansion.
- It enables iterative AI review once deterministic harness exists.

---

## 6. How to work this backlog

For each task:

1. Link affected fidelity IDs (for example, `T4-1 -> FC-017, FC-018`).
2. Implement and add/update scenario tests.
3. Mark checklist status (`NOT STARTED` -> `PARTIAL` -> `DONE`) with proof note.
4. Capture any deliberate deviation in a "known differences" section.

This keeps progress measurable and prevents fidelity regressions.
