# 8.0 Fidelity Checklist & Epic Backlog

## 1. Purpose

This document defines the delivery plan to make this multiplayer game match the feel and rules of **Digimon Digital Card Battle (PS1)** as closely as possible.

It has two goals:

1. Provide a **fidelity checklist** that can be reviewed as pass/fail.
2. Break the work into **epics and tasks** that can be completed one by one.

---

## 2. Canonical references (source of truth)

Use these references in priority order when deciding behavior:

1. PS1 manual + verified gameplay behavior (battle flow, support/effect rules, phase order).
2. Card-level behavior references (card text/effect catalogs), including the complete card-list reference:  
   `https://gamefaqs.gamespot.com/ps/526754-digimon-digital-card-battle/faqs/78563/100-card-collection-extra-notes`
3. Multi-match gameplay videos for pacing and UX timing validation.

Notes:

- A single video is not enough for mechanical parity.
- When references disagree, document the chosen behavior in the task that implements it.

## 6. How to work this backlog

For each task:

1. Link affected fidelity IDs (for example, `T4-1 -> FC-017, FC-018`) or GDD phase tasks (for example, `P2-3 -> FC-006`).
2. Implement and add/update scenario tests (FC) or manual/video validation notes (UX).
3. Mark checklist status (`NOT STARTED` -> `PARTIAL` -> `DONE`) with proof note.
4. Capture any deliberate deviation in a "known differences" section.
5. Regenerate `fidelity-report.md` via `pnpm fidelity:report` when adding FC-mapped scenarios.
6. Update §3.5 phase rollout status when a GDD phase slice is signed off.

This keeps progress measurable and prevents fidelity regressions.
