# **Database Schema & C3 Integration Guide**

## **1\. Executive Summary: The "Data-Driven" Game**

To ensure maximum performance and scalability, the game engine must remain agnostic to the cards. Construct 3 will never ask "Is this Agumon?"; it will only ask "What is the hp value at Cards\[0\]?". This allows us to balance the game instantly just by tweaking a JSON file, without ever compiling or editing the C3 Event Sheet.

## **2\. Core Mechanics: The Schema Breakdown**

### **A. The Evolution Economy**

* dp\_yield: The amount of DP added to the Global Variable PlayerDP when this card is selected and sent to the Discard Pile.  
* evo\_cost: The amount subtracted from PlayerDP to play this card onto the Active Field. (Rookies are always 0).

### **B. The Combat Variables**

* atk\_o, atk\_t, atk\_x: Base stats for the calculation pipeline.  
* x\_effect: A string ID (e.g., "counter\_o", "x\_to\_zero"). The CalculateDamage function uses a Switch statement on this string to apply special conditional math before resolving damage.

### **C. The Support System**

* support\_id: A string ID (e.g., "buff\_o\_200"). When a card is placed face-down in the Support Phase, the engine reads *this* field, ignoring HP and Attack stats.

## **3\. Technical Implementation (C3 Specific)**

### **A. Loading the JSON Array**

1. Add the **JSON object** to the project.  
2. On Start of Layout:  
   * Load the Starter\_Deck.json using the AJAX plugin.  
   * Action: JSON \-\> Parse string \-\> AJAX.LastData.

### **B. Spawning the Static Deck (The Hand)**

Because we are using a static deck for MVP testing, we don't need a complex shuffle algorithm yet.

1. Create a global array Player\_Deck.  
2. Loop through the JSON array (JSON.Get("cards.length")).  
3. Push each id into Player\_Deck.  
4. When drawing a card, spawn Fam\_Cards and set its internal CardID string.  
5. Call Function UpdateCardVisuals(InstanceID), which looks up the JSON using the CardID to populate the pinned Text objects (Name, HP, Stats).

### **C. The "Support Dictionary"**

Do not write 100 If statements for support cards. Use the **Dictionary Behavior**.

* When support\_id \== "buff\_o\_200":  
  * Action: Set Tmp\_P1\_O to Tmp\_P1\_O \+ 200\.  
* When support\_id \== "swap\_enemy\_o\_to\_t":  
  * Action: Set Tmp\_P2\_Selection to 1 (Triangle).

## **4\. Optimization Tip**

Never load JSON data directly inside an "Every Tick" event. Store current battle data (Active HP, Active Atk values) in Global Variables or Instance Variables on the OBJ\_BattleBrain during the "Draw/Evolution Phase." The math should only calculate using those temporary variables, minimizing CPU load and DOM lookups.
