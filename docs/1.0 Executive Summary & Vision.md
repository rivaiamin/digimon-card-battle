**GDD: Project Digital Arena (Working Title)**

## **1\. Executive Summary**

**The Elevator Pitch:** A strategic card-battler where players evolve monsters in real-time during a match to overcome elemental disadvantages. It’s a faithful recreation of the PS1 classic's mechanics—triangular RPS combat, support card layers, and DP-based evolution—wrapped in a modernized, high-fidelity UI.

**The North Star:** "Tactical Evolution." Every turn must feel like a gamble between immediate power (Support cards) and long-term scaling (Evolution).

## **2\. Core Mechanics (The "DCA" Logic)**

* **The Trinity Attack System:** Each Digimon has three attacks: **O** (Power), **Triangle** (Speed/Utility), and **X** (Counter/Ability).  
* **DP (Digimon Points) & Evolution:** Evolution is not free. Players must discard cards to gain DP or meet specific requirements to move from Rookie → Champion → Ultimate.  
* **The Support Phase:** Before attacks resolve, players play one card face down. This is the "hidden variable" that shifts the tide (e.g., "Attack x2" or "Opponent's attack becomes 0").  
* **Win Condition:** Reduce opponent's HP to 0 or deplete their deck (Deck Out).

## **3\. The 2.5D Framework**

In C3, we will simulate the 3D "Tabletop" feel of the original without the overhead of a 3D engine.

* **The Perspective:** We will use a **Mode 7 or 2.5D Isometric projection** for the Battle Arena.  
* **Y-Sorting:** For World Map navigation (Battle Cafes), we will use the Z-Order \> Sort by Y system.  
* **Z-Elevation:** Card hover effects will use a "Z-Elevation" variable mapped to the Scale and Shadowcaster behavior to create a lift-off effect from the table.

## **4\. Technical Implementation (C3 Specific)**

### **A. Recommended Plugins & Behaviors**

* **LiteTween / Tween (Built-in):** Essential for card draws, flips, and damage shakes.  
* **JSON:** The backbone. Every card is an entry in a JSON file.  
* **Dictionary:** To manage the "Current Match State" (Player HP, DP, Deck Count).  
* **Families:**  
  * Fam\_Cards: Contains variables like CardID, Type, Level, Attack\_O, SupportID.  
  * Fam\_UI\_Buttons: For consistent hover/click logic.

### **B. Data Management (The Card Engine)**

We will **not** create a separate sprite for every card.

1. **Card Template:** One Sprite with two animations (Front/Back).  
2. **Dynamic Loading:** Use the Binary Data or Project Files to load a JSON card database.  
3. **The Parser:** On "Card Created," the engine reads the CardID and updates the text objects pinned to the card (Name, HP, Attack Values) from the JSON.

### **C. UI/HUD Strategy**

* **Layer 0 (Background):** 2.5D Arena.  
* **Layer 1 (Game Surface):** The "Field" where cards sit. Uses a slight Parallax (e.g., 90%) to give depth.  
* **Layer 2 (HUD):** 100% Parallax. Modernized flat-design bars for HP and DP.  
* **Layer 3 (Modals):** For the "Support Card" reveal and "Evolution" animations.

**Engineering Documentation: Implementation Roadmap**

## **I. Data Schema (JSON Structure)**

The Engineer must follow this structure for the Cards.json:

JSON

{  
  "CardID": 001,  
  "Name": "Agumon",  
  "Level": "Rookie",  
  "Type": "Fire",  
  "HP": 400,  
  "Atk\_O": 200,  
  "Atk\_T": 140,  
  "Atk\_X": "Counter Fire",  
  "SupportID": "dmg\_mult\_2",  
  "EvolutionDP": 0  
}

## **II. Event Sheet Architecture**

To keep the project clean, use **Global Signals** for the Battle Flow:

1. **State: DrawPhase** → (Trigger: Signal "NextPhase")  
2. **State: EvolutionPhase** → (Logic: Check if Hand card Level \> Current Level)  
3. **State: SupportPhase** → (Input: Wait for Player/AI click)  
4. **State: BattlePhase** → (Calculation: Apply Support effect → Compare Atk vs Atk → Subtract HP)

## **III. Optimization Constraints**

* **Texture Memory:** Use **Sprite Sheets** for card art. Instead of 300 individual images, use a few 2048x2048 sheets.  
* **Event Cleanup:** Use "On Created" for card initialization and "Destroy" for discarded cards. Do not leave cards off-layout; use a RecycleBin family to toggle Object Visibility and Disabled state to save CPU.  
* **Z-Order Management:** Use a dedicated Z-Order system. Every frame, Fam\_Cards should be sorted by an instance variable DepthLayer to ensure the "Held Card" is always on top of the "Field Cards."

## **IV. The "Fresh UI" Directive**

* **Glow & Particles:** Use C3's Additive blend mode for "Evolution" glows.  
* **Glassmorphism:** Use the Blur effect on UI background panels to give a modern, premium feel while keeping the "Digimon Blue" color palette.
